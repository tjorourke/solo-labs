📋 How to drive this demo click to expand
1. Set the scene with the three panels above.
Left = what the registry says is approved · Middle = what the gateway lets through · Right = the full upstream surface. The middle should match the left exactly; the right has everything (legit + dangerous + poisoned + late-added).
2. Click the three green buttons (happy path) to prove the gateway is selective, not blanket-deny. Each returns a real tool result.
3. Click the four red buttons in order — each bypasses one enforcement layer: deny-by-default · args schema · risk-tier × intent · forbidden chain. Read the trace below each click — the deny message tells you which layer fired.
4. Live edit — registry as authority. In a terminal, edit the manifest: 
kubectl -n tool-curation edit configmap curation-manifest
Remove db_read_secret from approvedTools. Within ~5s, the middle panel updates and attack 3 now returns -32602 Unknown tool from the gateway (the ext-auth layer never sees it). No policy CR was edited by hand — the registry is the authority.
About the intent dropdown. The dropdown switches between two JWTs the in-cluster issuer minted at startup. Both JWTs have the same sub: agent. What differs is the intent claim — what the agent is doing, not who it is. See the JWT viewer below for the actual claims being sent. This is purpose-based access control (PBAC) — distinct from role-based (RBAC) or attribute-based (ABAC).
